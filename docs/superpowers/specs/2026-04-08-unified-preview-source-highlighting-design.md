# Unified Preview Source Highlighting Design

## Goal

让所有文档预览类型在从问答溯源进入时，都能稳定完成同一套闭环行为：自动定位到目标朔源块、以统一的块级高亮样式展示、支持点击高亮块跳转详情、并保留返回 AI 回答的路径。

## Why

当前预览定位能力不一致：

- PDF 依赖浏览器内置预览器，只能近似跳页，无法稳定实现正文块级高亮
- TXT / JSON 主要依赖字符串命中，只能做到词级或片段级高亮
- Excel / CSV 只能高亮单行，缺少统一的块级区域语义
- 各预览类型的定位策略与视觉表现不一致，用户在不同文件类型间切换时体验割裂

这会直接破坏“问答溯源可追、可看、可跳”的核心价值，因此需要统一为单一预览定位模型和统一高亮规范。

## Scope

本次范围只覆盖预览定位与高亮闭环，不扩展到新的文档类型或新的批注系统。

包含：

- PDF / TXT / DOCX 文本预览的块级高亮定位
- JSON 原文预览的节点级块高亮定位
- Excel / CSV 预览的 Sheet + 行区间块高亮定位
- 问答溯源到预览的统一定位数据结构
- 预览内高亮块点击跳详情

不包含：

- 详情页重构
- PDF 自由批注、框选、评论
- JSON 树形编辑器
- 新的路由体系

## UX Rules

所有预览类型统一遵守以下规则：

1. 从问答点击溯源后，预览必须自动滚动到目标块，而不是只停留在顶部或只切到目标页
2. 高亮必须是块级高亮，而不是只给命中的几个词加 `mark`
3. 高亮块统一使用以下视觉规范：
   - 背景：`#FFF7CC`
   - 边框：`1px solid #E8C95A`
   - 圆角：`8px`
   - 阴影：轻微强调，不压过正文
4. 高亮块本体可点击，行为等同于“跳转详情”
5. 顶部黄色定位条保留，但只作为辅助提示，不再替代正文高亮
6. 若无法精确命中原始块位置，允许降级到最接近的结构块，但必须明确高亮一个可见块，而不是静默失败

## Unified Target Model

现有 `sourceHighlight` 信息不足，只包含 `chunkId/chunkIndex/content`，容易导致模糊命中和重复命中。需要统一升级为“预览定位目标”模型，优先使用结构化定位，文本片段仅作兜底。

统一模型字段：

- 通用字段
  - `docId`
  - `chunkId`
  - `chunkIndex`
  - `content`
  - `originStart`
  - `originEnd`
- 文本文档字段
  - `pageStart`
  - `pageEnd`
  - `textQuote`
  - `textOffsetStart`
  - `textOffsetEnd`
- 表格字段
  - `sheetId`
  - `sheetName`
  - `rowStart`
  - `rowEnd`
  - `columnStart`
  - `columnEnd`
- JSON 字段
  - `jsonPath`
  - `nodeStartOffset`
  - `nodeEndOffset`

## Target Resolution Priority

所有渲染器统一按以下优先级解析定位目标：

1. 结构化定位
   - PDF / DOCX / TXT：页码、offset、quote
   - 表格：sheet + 行区间
   - JSON：jsonPath 或 offset
2. chunk 对应原文片段
3. 字符串模糊命中

这样可以显著降低重复内容导致的误定位风险，并为后续失效判断与端到端测试提供稳定契约。

## Rendering Strategy

### PDF

PDF 预览必须使用标准 `pdfjs` 页面结构，恢复到正常 PDF 阅读观感，不得使用浏览器原生 iframe，也不得退化为纯文本页或“看起来像 PDF 的替代壳层”。页面结构固定为三层：

- `canvas page`：负责渲染真正的 PDF 页面内容
- `text layer`：透明文本层，仅用于文本定位、选择和高亮锚点
- `overlay layer`：只承载轻量高亮覆盖与旁侧提示条

核心策略：

- 使用 `pdfjs-dist` 渲染真实页面到 `canvas`
- 在 `canvas` 上方叠加标准 `text layer`，避免把文本本身作为预览主体展示
- 根据 `pageStart/pageEnd + textQuote/offset` 找到最接近的正文范围
- 在正文原位置上叠加轻量半透明黄底高亮，尽量贴合原文块边界
- 在命中区域旁侧显示轻量提示条，说明该区域为命中的溯源块
- 自动滚动到目标块，并支持高亮块点击跳详情

PDF 的高亮优先级为“原文阅读感优先”。高亮和提示必须可见，但不能像整段荧光笔覆盖那样压住原 PDF 的阅读体验。任何为了测试或降级而引入的替代性展示层，都不能破坏“第一眼看上去就是正常 PDF 页面”的观感。

### TXT / DOCX

统一视为“连续文本块预览”。实现策略：

- 根据结构化 offset 优先切出目标块
- 无 offset 时，退化到 quote/content 命中
- 渲染为 `before / highlighted block / after`
- 高亮块整体可点击，不再只是对命中词做内联 `mark`

### JSON

JSON 继续保持原文文本视图，不切回树形结构。实现策略：

- 优先使用 `jsonPath` 找到目标节点范围
- 无 `jsonPath` 时使用 offset，再退化到文本片段匹配
- 将目标节点整个文本片段视为一个块级高亮区域
- 保持原文阅读体验，不插入额外树结构控件

### Excel / CSV

表格预览继续基于 Sheet 维度展示，但高亮从“单行命中”升级为“块区间命中”。实现策略：

- 根据 `sheetId/sheetName` 优先定位到目标 Sheet
- 使用 `rowStart/rowEnd` 高亮整段行区间
- 若只能模糊匹配，则回退到最接近的行集合
- 高亮区块应有统一样式，并支持点击进入详情

## Component Boundaries

建议增加统一的预览定位工具层，而不是把各类匹配逻辑散落到各个渲染器内部：

- 统一目标类型定义：负责承载结构化定位信息
- 统一目标解析器：把来源信息映射到各渲染器可消费的数据
- 统一高亮样式组件：保证视觉一致
- 各渲染器仅负责“本格式如何滚动到目标块并渲染块级高亮”

这样可以避免后续继续出现 PDF 一套、JSON 一套、表格一套的割裂实现。

## Error Handling

- 若目标文档已失效：沿用现有“已失效”兜底，不打开预览正文高亮
- 若结构化定位信息缺失：回退到文本匹配
- 若文本匹配失败：至少滚动到最接近页 / Sheet / 节点容器，并显示顶部定位条，不允许无反馈
- 若 PDF 页级定位成功但文本块匹配失败：优先保持标准 `pdfjs` 页面视觉，滚动到目标页，并在页内以轻量提示条说明“已定位到目标页，正文块匹配已降级”，不得回退为纯文本页、iframe 搜索态或其它替代性页面壳层

## Testing Strategy

需要补齐以下测试层级：

- 预览定位模型解析单测
- PDF / Text / JSON / Table 各自的块级高亮渲染单测
- `DocumentPreviewContent` 集成测试，验证统一目标能传递到对应渲染器
- `DocumentListPanel` / `App` 级联测试，验证从问答点击溯源后真的打开预览并定位到目标块

## Acceptance Criteria

以下条件全部满足后，视为实现完成：

1. PDF / TXT / JSON / Excel / CSV 五类预览都能自动滚动到目标块
2. PDF 保持原版页面视觉，其它预览保持各自原文视图；所有类型都使用统一的块级高亮语义规范
3. 高亮块本体可点击跳转详情
4. 顶部定位条保留，但正文高亮始终可见
5. 任一预览类型在结构化定位缺失时，仍能退化到可见高亮，而不是静默失败
6. 从问答点击溯源进入预览的测试链路全部通过
